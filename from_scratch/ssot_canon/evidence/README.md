# Evidence Folder Guide

_Status: draft_
_Last updated (UTC): 2026-04-08_
Canonical terminology: ../10_product_and_architecture/CANONICAL_TERMINOLOGY.md

## Purpose
Describe evidence artifact expectations and storage layout for SSOT verification and release readiness.

## Scope
Evidence files generated by testing, smoke checks, and docs sync automation.

## Normative statements
- SSOT-impacting releases MUST attach evidence artifacts under a dated release folder.
- Evidence artifacts SHOULD include command, timestamp, environment, and outcome.
- Sensitive data MAY NOT be committed in evidence logs.

## Interfaces / contracts
### Suggested structure
- `evidence/releases/YYYY-MM-DD/`
- `evidence/ssot-sync/YYYY-MM-DD/`
- `evidence/templates/*`

Templates are provided in `evidence/templates/`.

## Failure/rejection semantics
- Missing evidence for required gates can block release approval.
- Redaction failures in evidence logs are security incidents.

## Verification requirements
- Release and QA reviewers validate evidence completeness.

## Traceability hooks
- Code refs: smoke scripts under `scripts/`
- Tests refs: contract/security suite outputs
- Related SSOT docs: `../40_operations_and_quality/OPERATIONAL_SMOKE_CHECK_CONTRACT.md`, `../40_operations_and_quality/RELEASE_CHECKLIST.md`
